# ADR-0003: 경로 탐색을 외부 API 오케스트레이션 서비스로 분리

- 상태: Accepted (2026-06-18 feasibility 검증 후 개정)
- 날짜: 2026-06-18
- 관련: [feasibility.md](../../feasibility.md)

## 맥락

feasibility 검증 결과 두 가지가 확인되었다.

1. **도로 경로 자체는 상용 API로 완전히 해결된다.** 네이버 Directions / 카카오모빌리티 /
   TMAP가 자동차 경로·다중 경유지·통행료·소요시간을 제공한다. 직접 도로망 라우팅 엔진
   (OSRM/Valhalla)을 운영할 필요가 없다.
2. **그러나 "배터리 잔량 기반 충전 정차 자동 삽입"은 어떤 상용 API도 제공하지 않는다.**
   이 로직은 직접 구현해야 한다.

따라서 우리의 핵심 작업은 *도로 그래프 탐색*이 아니라, *외부 경로 API를 도구로 호출하면서
후보 충전소를 평가·선택하는 오케스트레이션*이다. 이 작업은 다수 충전소 후보에 대한 반복
계산(경로 매트릭스 호출 포함)이 필요해 Supabase Edge Function 제약에 부적합하다.

## 결정

`apps/routing-service`를 별도 서비스로 둔다(Node.js/TypeScript). **자체 도로 엔진은 두지 않는다.**

- **도로 경로/거리/시간**은 외부 경로 API(네이버·카카오·TMAP 중 택1, 약관·쿼터 비교 후 결정)에
  위임. provider를 adapter로 추상화해 교체 가능하게 한다.
- 후보 충전소 간 거리·시간 일괄 계산은 **경로 매트릭스(Route Matrix) API**를 활용한다.
- **충전 정차 최적화 알고리즘**은 `packages/core`의 순수 함수로 두고, 서비스는 외부 API
  호출과 결과 조합(I/O)만 담당한다.
- 충전소 데이터는 환경부 API를 Supabase로 동기화한 **자체 캐시 DB**에서 조회한다
  (외부 쿼터 보호, ADR-0002 / feasibility §5 참고).
- BFF가 이 서비스를 호출하고 결과를 조합한다.

## 결과

- 장점: 도로 엔진 운영 부담 제거, provider 교체 가능, 순수 코어로 단위 테스트 용이.
- 단점: 외부 경로 API 약관·쿼터·과금에 종속(미해소 리스크, feasibility §5).
- 위험 완화: provider adapter 추상화 + 경로 매트릭스 호출 수 최소화(후보 충전소 사전 필터링) +
  결과 캐싱.
